# 微博平台城市洪水内涝信息爬取框架开发方案

## 1. 文档目的与范围
- 目标：基于《White_Paper.md》的“采集-清洗-多模态抽取-时空融合”架构，落地微博平台的城市洪水内涝信息爬取与结构化处理框架。
- 范围：覆盖历史数据采集、清洗过滤、结构化抽取、地理融合、数据服务与质量评估。
- 非目标：不在本阶段实现全平台（多社媒）统一接入；不在本阶段建设复杂前端可视化系统。

## 2. 业务目标与核心指标
- 业务目标
  - 基于关键词和城市范围，一次性批量采集历史微博洪涝信息。
  - 自动识别“真实内涝事件”并输出结构化事件记录。
  - 为后续遥感融合与灾害复盘研究提供可分析数据源。
- 核心指标（首版建议）
  - 历史采集完整性：目标时间窗内关键词命中微博抓取成功率 >= 90%。
  - 事件识别准确率：洪涝相关性分类 F1 >= 0.85。
  - 地理定位可用率：可落点事件占比 >= 70%。
  - 去重率：重复微博/转发内容识别率 >= 90%。
  - 批处理稳定性：单次批处理任务成功率 >= 95%。

## 3. 总体架构设计
- 分层架构
  - 接入层（Ingestion）：微博检索页/话题页/用户页历史抓取，批处理任务调度。
  - 处理层（Processing）：清洗、去重、相关性分类、结构化抽取。
  - 融合层（Fusion）：地理编码、时空归一化、可信度评分。
  - 服务层（Serving）：事件 API 与数据导出。
  - 运维层（Ops）：任务监控、日志审计、任务回放。
- 技术形态
  - 爬虫引擎：`Crawl4AI` + Playwright（动态渲染）+ 代理池。
  - 存储：
    - 原始层：对象存储（HTML/Markdown/图片元数据）。
    - 明细层：PostgreSQL（结构化微博记录）。
    - 检索层：Elasticsearch（全文检索与分析）。
  - 算法服务：LLM 分类/抽取服务、图像二分类服务。
  - 编排：Airflow 或 Celery Beat + Worker。

## 4. 微博采集框架设计

### 4.1 数据源与入口策略
- 关键词检索入口：如“内涝/积水/道路被淹/地下车库进水/暴雨被困”。
- 城市定向入口：`城市名 + 洪涝词`组合查询（如“广州 积水”）。
- 事件话题入口：`#暴雨#`、`#城市内涝#`等历史话题检索。
- 高价值账号入口：气象、应急、交警、城运、媒体、社区账号。

### 4.2 批量采集执行策略
- 关键词设计：按“城市词 + 灾情词 + 场景词”组合生成检索式。
- 时间分片：按月或按周拆分历史时间窗，避免单任务跨度过大。
- 任务组织：一次性运行批量采集，按“城市-关键词-时间片”三维任务并发执行。
- 断点续跑：记录每个任务的游标和状态，失败后仅重跑失败分片。

### 4.3 反爬与稳健性策略
- 指纹与节流：随机 UA、请求间隔抖动、并发上限控制。
- 代理路由：住宅代理/机房代理混合池，失败自动切换。
- 重试策略：指数退避 + 熔断机制（避免连续触发风控）。
- 任务幂等：以 `platform + post_id` 作为主键去重，防止重复写入。

### 4.4 数据落地规范
- 原始记录字段
  - `platform`, `post_id`, `author_id`, `author_name`, `publish_time_raw`, `crawl_time`
  - `text_raw`, `media_urls`, `repost_count`, `comment_count`, `like_count`
  - `source_url`, `search_keyword`, `topic`, `city_hint`
- 标准化记录字段
  - `publish_time_std`, `city_std`, `district_std`, `event_type`, `flood_relevance_score`
  - `water_depth_cm`, `road_blocked`, `people_trapped`, `geo_point`, `confidence_score`

## 5. 数据清洗与过滤流程

### 5.1 预处理
- 文本清洗：去 HTML 标签、表情转义、URL/短链标准化。
- 语言检测：过滤非中文或低信息密度文本。
- 媒体处理：下载元数据、校验可访问性、抽取图像 hash。

### 5.2 去重体系
- 文本去重：SimHash/MinHash 双轨；阈值按城市事件密度动态调节。
- 转发聚合：识别原帖-转发链，仅保留主帖 + 统计传播强度。
- 图像去重：pHash + 感知阈值聚类。

### 5.3 相关性分类
- 分类标签
  - `flood_fact`（事实性灾情）
  - `help_request`（求助）
  - `traffic_impact`（交通影响）
  - `emotion_or_metaphor`（情绪/比喻）
  - `rumor_risk`（疑似谣言/导流）
- 流程
  - 轻量模型初筛（高召回）。
  - LLM 复判（高精度）并输出判定理由。

## 6. 多模态信息抽取方案

### 6.1 文本结构化抽取（Schema First）
- 使用 `Pydantic` 定义强约束 Schema：
  - 时间：事件发生时间、发布时间、时间可信度。
  - 地点：POI、道路、交叉口、小区、地铁站等。
  - 灾情：积水深度、影响对象、通行状态、人员风险。
- 通过 `Instructor` 或函数调用模式强制 JSON 输出，失败进入重试队列。

### 6.2 图像验证（Few-shot 二分类）
- 目标：判断图像是否为“洪涝现场”。
- 数据：首期人工标注小样本（每类 200-500 张）建立基线。
- 模型：选择 5-6 个少样本分类模型对比（如 ProtoNet、MatchingNet 等）。
- 融合：图像结果作为文本事件置信度增益项，不单独成事件。

## 7. 时空融合与事件生成

### 7.1 地理编码与归一
- 地理编码：高德/百度 API 将地点描述映射为坐标。
- 坐标统一：统一到 WGS84，保留原始坐标系字段。
- 模糊落点：针对“某路口/某车库”进行候选点打分。

### 7.2 事件聚合
- 聚合维度：`城市 + 1km 网格 + 30分钟窗口 + 事件类型`。
- 聚合输出：
  - `event_id`, `start_time`, `end_time`, `event_level`
  - `evidence_count`, `help_request_count`, `median_water_depth_cm`
  - `event_confidence`, `top_evidence_posts`

### 7.3 可信度评分
- 评分因子：账号可信度、文本分类置信度、图像验证结果、时空一致性、跨源一致性。
- 评分区间：0-1，建议阈值
  - `>=0.8` 高可信
  - `0.5-0.8` 待核验
  - `<0.5` 低可信

## 8. 服务接口与数据产品
- 事件查询 API：按城市、时间、事件等级过滤。
- 数据导出：CSV/GeoJSON，用于遥感验证和报告生产。
- 审核台（可后置）：人工复核低可信事件并反哺训练集。

## 9. 研发实施计划（10周）

### 阶段 0：需求冻结与技术验证（第 1-2 周）
- 输出
  - 关键词词库 v1、城市名单、账号白名单。
  - 微博采集 PoC（单城市、单关键词）。
  - 合规与风控边界确认文档。
- 验收
  - 单次历史采集任务可稳定完成并可复跑。

### 阶段 1：批量采集框架开发（第 3-4 周）
- 输出
  - 批处理调度器、分片任务管理、重试与代理模块。
  - 原始数据入库与任务日志体系。
- 验收
  - 历史分片任务可并发执行，失败任务可断点续跑。

### 阶段 2：清洗与抽取流水线（第 5-7 周）
- 输出
  - 去重模块、相关性分类模块、Schema 抽取模块。
  - 图像二分类 baseline。
- 验收
  - 抽样评估达到 F1 与去重率目标的 80% 以上。

### 阶段 3：时空融合与服务化（第 8-9 周）
- 输出
  - 地理编码服务、事件聚合引擎、事件 API。
- 验收
  - 关键城市事件可在地图网格维度稳定聚合。

### 阶段 4：联调、验收与交付（第 10 周）
- 输出
  - 压测报告、任务回放方案、运行手册。
- 验收
  - 关键链路可追踪、可复跑、可导出。

## 10. 测试与质量保障
- 测试分层
  - 单元测试：解析器、去重、时间地点归一化。
  - 集成测试：采集 -> 清洗 -> 抽取 -> 融合全链路。
  - 回归测试：提示词、分类阈值变更后的效果对比。
- 数据评估
  - 每周人工抽样 300-500 条，评估分类与抽取精度。
  - 建立错误样例库（误报/漏报/定位失败）持续迭代。

## 11. 监控与运行保障
- 运行监控
  - 任务成功率、失败分片数、任务耗时、API 成功率。
- 质量监控
  - 日新增事件数、城市覆盖率、置信度分布漂移。

## 12. 合规与安全要求
- 严格遵守平台协议与法律法规，限制采集频率和使用范围。
- 敏感字段脱敏存储（用户标识可哈希化）。
- 日志审计与访问控制（最小权限原则）。
- 模型输出保留可追溯证据链（原文片段 + 判定理由）。

## 13. 风险清单与应对
- 平台策略变动导致采集不稳定
  - 应对：多入口冗余 + 采集策略热更新。
- 语义歧义导致误报
  - 应对：分层分类 + 人工复核回路。
- 地点描述模糊导致定位偏差
  - 应对：候选点排序 + 多证据联合修正。
- 突发峰值导致系统拥塞
  - 应对：弹性扩容 + 任务优先级降载。

## 14. 交付物清单
- 架构设计文档、接口文档、数据库字典。
- 采集框架代码与部署脚本。
- 分类与抽取模型配置、评估报告。
- 任务监控面板、运行手册。
- 阶段复盘与下一阶段优化路线图。

## 15. 下一步建议
- 增加跨平台联动（短视频平台/本地论坛）提升事件召回。
- 引入遥感积水产品进行自动交叉验证。
- 建设主动学习闭环，降低人工标注成本并持续提升精度。
